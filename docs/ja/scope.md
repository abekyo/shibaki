# Shibaki のスコープ (受理するタスク)

受理判定は **「完遂に使える客観シグナルが存在するか」** のみで切る。
主観が混じった瞬間 critic が無力化し、Shibaki の North Star が崩壊する。

---

## Tier 1: 受理する

**定義**: 「できた」をコマンド 1 発の exit code で判定できるタスク

| カテゴリ | 例 | verify コマンド例 |
|---|---|---|
| failing test 修正 | 「このテストを通して」 | `bun test path/to/test.ts` |
| 型エラー解消 | 「型エラーを消して」 | `tsc --noEmit` |
| lint / format | 「eslint 違反直して」 | `eslint src/ --max-warnings 0` |
| ビルド通し | 「ビルド通して」 | `bun run build` |
| 任意スクリプト | 「このスクリプトが exit 0 返すまで」 | `./scripts/check.sh` |

**critic の仕事**: そのシグナルを満たす過程で **ズル (test skip, `@ts-ignore`, mock 偽装)**
をしていないか反例付きで詰める。

---

## Tier 2: 将来検討

**定義**: 成功基準をユーザーが明示入力できるタスク

- リファクタ (「挙動を変えずに X を Y に置換」+ 既存テスト全通過)
- 新規機能 (受け入れ基準を `--accept` で渡せた場合のみ)

現在は受理しない。ユーザーに受け入れ基準を書かせるのは設定項目の増加に近く、
プロダクトの設計原則 (シンプル UX) と緊張関係にある。Tier 1 が浸透してから検討する。

---

## Tier 3: 永久に受理しない

**定義**: 成功が主観 / 文脈依存のタスク

- 設計判断 (「どのライブラリ使うべき?」)
- ドキュメント / コメント生成
- エラー文言の推敲
- 「コードを綺麗にして」等の曖昧命令
- UI の見た目調整
- コードレビュー (「これどう思う?」)

**理由**: critic が反例を出せない → 結論を出すために critic ログを
人間が読む羽目になる → 「人間をループに入れない」原則違反。

**対応**: Shibaki は受理拒否し、素のエージェントに誘導する。
