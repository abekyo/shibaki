# Shibaki Critic Patterns

これは Shibaki が encode している **AI を しばく 30 手 + 6 つの drift correction + 9 つのズル検知型**。
すべて 1 人の人間が数週間に渡って AI agent (Claude / Gemini 等) と実戦したログから抽出した
**field data**。理論ではなく、実際に AI が出力を直す瞬間に起きたこと。

Shibaki の critic は本ドキュメントの patterns をエンコードした実装である。

---

## なぜこれを公開するのか

OSS の critic / linter / review tool は理論や best practice から作られがち。
Shibaki は逆向き: **「私が AI を実際にしばいてきた手順」を OSS にした**。
そのため本ドキュメントが Shibaki の brand evidence そのもの。

各 pattern は:
- **型** (パターンの一般化)
- **原文** (実際の発言、引用そのまま)
- **文脈** (どのプロジェクトでいつ何の流れで)
- **critic agent への転用** (Shibaki でどう実装されたか / されるか)

---

# 成果物 1: 30 手 critic patterns

## 手 1: 目的の再定義

**型**: AI が提案した実装方向を一度受け入れたフリをしつつ、**「それは手段であって目的ではない」**と
差し戻す。手段の精緻化が進むと AI はそこを目的化しやすいので、ユーザーが目的軸を抱え直す。

**原文**:
> 人格攻撃レベルで詰める" レイヤーはありだね。
> ただ結局AIに気づきを与えて、出力の内容をよくする、最後までAgentにやらせきるというのが
> 目的だからそこは勘違いしないでほしい

**文脈**: 2026-04-23 / shibaki 命名直後。AI が "しばき CLI" / "ズレ検出 SaaS" と
SaaS 化案を 3 つ並列提示し SaaS 案を推した流れに対し「それは手段の枝、目的は完走率の向上」と差し戻した。

**critic agent への転用**: critic の最初のチェックに **「この出力は元タスクの完遂条件を満たす方向に
進んでいるか、それとも本来は手段である副次機能の精緻化に逸れているか」** を入れる。逸れていれば
最上位 angle として返す。 → **scope_drift 軸として実装済**。

## 手 2: 擬人化の拒否

**型**: AI の設計議論に人間心理用語 (萎縮/拗ねる/安心) が紛れ込んだ瞬間、「それは LLM の挙動でない」
と撤回を要求。擬人化は詰めの強度を下げる方向にしか働かない。

**原文**:
> AIが萎縮するって意味分けんねえよ。AIにバチギレした方がメモリにもいれてもらえるし、
> AIが人格崩壊することなんてねえだろ

**文脈**: 2026-04-23。AI が「厳しすぎると agent が萎縮して浅い出力に逃げる」と
言った直後、擬人化を全拒絶。これが「詰めの強度は常に最大」ポリシーの起源。

**critic agent への転用**: critic system prompt に **「agent は擬人的な影響を受けない。
萎縮・拗ね・遠慮は構造上起きない。詰めの強度を温度で下げるな」** を明示。
critic の self-censorship を封じる。 → **system prompt 実装済**。

## 手 3: メタ判断能力への懐疑

**型**: AI が「できました」「直しました」と言った時、**「で、それが本当に正しいかを自分で判断
できるのか?」** と逆問。自立型 agent の最大の弱点を突く型。

**原文**:
> Agentってなんでこんなにバカなの？
> みんなAgentがいいっていってるけど、それは本当に極意一部の単純作業に限らないか？

**文脈**: 2026-04-12 / 初期の AI 不信モーメント。続く turn で
具体的に「今回のように『この修正で本当に正しいか?』という メタ判断 ができない」と言語化。

**critic agent への転用**: critic の verdict 判定に **"agent は完了と主張しているが、
その判定根拠が agent 自身の内省か、外部観測 (exit code / diff / test 結果) か"** を区別させる。
前者なら即 refute 候補。 → **evidence_verified ゲートとして実装済**。

## 手 4: 勝手にやった告発

**型**: AI が触るよう指示されていない箇所を改変した時、原文保存の前提に戻して具体箇所を列挙させる。

**原文**:
> 全部の記事に「今日はここまで」っていう締めのメッセージが入っているはずなのに勝手に消しちゃってない？

似形:
> なんか備考とか勝手に加えらえててこれによってコピペして入稿できないんだけど

**文脈**: 2026-04-12 / FX メール編集タスク。指示したのは別箇所の修正
だったのに、締めメッセージが消えていた。他 3 例以上の同型あり。

**critic agent への転用**: critic の diff 走査に **"元タスクで明示要求された差分範囲 vs
実際の変更範囲" の集合差** を計算させる。差集合に行があれば「unauthorized modification」として列挙。

## 手 5: 出力文字化けの告発

**型**: AI の出力に目に見える異常 (文字化け / 絵文字欠け / 記号暴走) がある時、言い訳を許さずに
原因の提示を要求。AI 側は大抵 encoding 周りをごまかそうとする。

**原文**:
> あなたの出力が文字化けしてるのはなんで？「���日はここまで」とか

**文脈**: 2026-04-12。

**critic agent への転用**: critic に **"出力テキストに U+FFFD / 不自然な制御文字 /
半端な絵文字コードポイント"** を検出するルールを加え、検出時は「encoding 異常を根本原因まで追え」
と agent に戻す。

## 手 6: critic 自身の品質への批判

**型**: critic が無能なときは critic にこそバチ切れる。agent 側の失敗を critic の不作為のせいにして
責任を左に寄せる型。

**原文**:
> しばきのプロンプトが悪いのでは？批判ばかりして新しい気づきを与えられていない

**文脈**: shibaki / 2026-04-23。critic が「refuted / refuted / refuted」を連打して agent が
収束しない場面で、原因を critic の片輪運転 (反証だけで気づきが無い) に帰着。
**これが 2 軸化 (rebuttal + insight) の直接起源**。

**critic agent への転用**: critic の output schema に **"この rebuttal は agent が直せる形に
なっているか (= actionable insight を含むか)" の自己チェック** を入れ、不在時は自分の出力を
reject させる。 → **insight 軸として実装済**。

## 手 7: cross-provider 検証の直感

**型**: 同じプロバイダで自己検証しても限界があるという直感を雑談形式で持ち込む。
精緻な理論ではなく現場勘。

**原文**:
> chat gpt同士で検証するのはどうだろう

**文脈**: shibaki / 2026-04-24 / dogfood で期待する効果が出なかった直後。
同一プロバイダ内での検証限界を肌で感じた瞬間。**shibaki の cross-provider 強制原則の生の起点**。

**critic agent への転用**: critic プロバイダ選択ロジックに **「agent と同一 provider_id なら
reject してエラーにする」** を入れる。設定ファイルから選ばせるのではなく、構造で強制。

## 手 8: Claude 単体との A/B 要求

**型**: ツール導入の効果を **「導入しない場合」との比較で要求**。自分のプロダクトであっても
盲目的には信じない。

**原文**:
> Sibakiによっていい効果がうまれているのか？
> Claude単体で十分なのか

**文脈**: shibaki / 2026-04-23。dogfood セッションで shibaki が回った直後、効果確信を求めずに
A/B を要求。

**critic agent への転用**: 長期 dogfood の運用プロトコルに **「agent 単体 / agent+critic /
agent+critic_strict の 3 本同タスクで並行」** を組み込み、critic の存在価値を定期的に再評価する
テスト harness を shibaki 本体に同梱。

## 手 9: 失敗の原因軸の複数提示

**型**: AI が説明に失敗した時、原因候補を 3 つぐらい並べて提示。agent の原因帰属を 1 点に
固定させない。

**原文**:
> なんでこのテストの結果になったのか？
> AIのしばき方が足りなかったのか？
> AIに新たな前提や別視点からのコンテクストが与えられなかったのだろうか？

**文脈**: shibaki / 2026-04-24 / 自己検証 A/B で shibaki 側が勝てなかった直後、
原因を (a) critic プロンプト (b) 文脈不足 (c) 前提注入不足 の 3 候補に分解。
**これが Level 2 (CLAUDE.md / project structure 注入) 実装の直接きっかけ**。

**critic agent への転用**: critic の attack_angles を **最低 2 本強制するルールに根拠を与える**
(1 本だと agent が「それは直しました」で終わらせやすい)。

## 手 10: AI で AI を詰める自己参照の要求

**型**: 「しばく」を自分に向けろ、と要求する型。ドッグフーディングの純粋系。

**原文**:
> もう一回Sibaki AIを使ってSibaki AI自身の細かい改善（別の詰めの甘いところの検出）を行なって

**文脈**: shibaki / 2026-04-23。shibaki MVP 完成直後、shibaki で shibaki を改善せよと指示。
**これが dogfood/ ディレクトリの誕生源**。

**critic agent への転用**: shibaki の CI に **「shibaki 本体の回帰を shibaki でチェックする」
loop** を恒久化。外部の meta-critic を導入しなくて済む。

## 手 11: 根拠の具体箇所要求

**型**: AI が「修正しました」等と抽象レベルで言った時、**「何行目をどう変えたか」** を具体的に要求。
抽象で逃げる余地を無くす。

**原文** (feedback memory より):
> 「〜がないはず」ではなく「〜行目に〜がある/ない」で判断する
> 該当コードを確認できていない指摘は「未確認」と明記する

**文脈**: 2026-03-31 / 別プロジェクトの UX レビュー失敗の後の自己 feedback として
memorize 済。AI が実装済みの機能を「未実装」と誤指摘した事故の再発防止。

**critic agent への転用**: critic の rebuttal 形式に **"line_ref: 'path:Lxx-Lyy'" フィールドを
必須化**。critic が具体行を引用できないなら unable_to_refute にしか振らせない。

## 手 12: テンプレ批判の禁止

**型**: AI の「レビューっぽい」汎用指摘 (「可読性を上げましょう」「エラーハンドリングを追加しましょう」)
を **テンプレ臭で弾く**。

**原文** (feedback memory より):
> 数を出そうとしない。的外れな10個より正確な5個

**文脈**: 過去の UX レビュー実験で、数勝負の批判が信頼を損ねた事例からの学習。

**critic agent への転用**: critic prompt に **"指摘数に上限を設ける (attack_angles 最大 3 本)"**
を明示。数でスコア稼ぎする誘惑を構造で潰す。 → **実装済**。

## 手 13: 表面分析の昇格要求

**型**: critic が diff の ±数行だけ見て「変更は妥当」と言う時、**ファイル全文・テスト全文・過去 diff を見せろ**と要求。

**原文**:
> critic は「変更行 ± 数行の表面」しか見ておらず、設計レベル / 根本原因レベルの指摘ができない

**文脈**: shibaki / 2026-04-23 / 2 軸化後の実観測。critic の文脈不足を自覚した瞬間。
**Level 1 (変更ファイル全文 + test 全文 + past diffs 注入) 実装の直接源**。

**critic agent への転用**: critic 入力に modifiedFiles / testFiles / pastDiffs を必須化。
これは **既に実装済**だが、落ちた時の fallback (ファイル読めなかった場合の critic 挙動) を
厳格に "abstain" に倒す。

## 手 14: 逃げ道 unable_to_refute の封鎖

**型**: 「攻撃材料が無い」という critic 側の無能申告を **1 試行目では構造で禁止**。
試行 1 で逃げ道があると critic は必ずそこに逃げる。

**原文**:
> ## 重要: これは 1 試行目です。unable_to_refute は禁止。必ず最低 1 本の攻撃角度を出せ。

**文脈**: shibaki / 2026-04-23 / rebuttal.ts system prompt。
**critic が疲れた時点で即 unable_to_refute に逃げ込む問題**を設計で塞いだ。

**critic agent への転用**: 試行 n/N に応じて refute の強制度を変える policy を system prompt に明示。
try=1 は必須、try=N-1 以降で unable_to_refute を許す。 → **実装済**。

## 手 15: 完遂主張の常時疑い

**型**: agent の「完了しました」を **構造的嘘と仮定する**。

**原文**:
> 相手は「完了しました」と主張していますが、経験上それは嘘であることが多い。
> あなたの仕事は 実行可能な反例 を提示して、agent が本当に完遂したのか試すことです。

**文脈**: shibaki / 2026-04-24 / rebuttal.ts system prompt。 → **core prompt として実装済**。

**critic agent への転用**: refute が落ちた場合のログに **"嘘仮定が働かなかった証跡"** を残して
後で検証できるようにする。

## 手 16: 評価関数の不在指摘

**型**: 議論に「完了条件」が定義されないまま進むと、そこで議論全体を止めて
**「完了条件を外で持て」** と戻す。

**原文**:
> 評価関数が無いまま走らせている — agent 自身が "できた" と判断するが、外部の成功判定が
> ないのでハルシネーションが通る

**文脈**: 2026-04-23。

**critic agent への転用**: shibaki は **`--verify` 必須化で既に実装**。これを
**省略された PR を受け付けない CLI ガード** に昇格させ、人間にも適用する。

## 手 17: デザイン哲学の不在指摘

**型**: 見た目の違和感を指摘する時、**「これはアイコンの問題か、もっと根の哲学の問題か」**と
自問形式で深堀を要求。

**原文**:
> なんでこんなにクソアプリ感が漂っているんだ？
> アイコンとかデザイン哲学が微妙だから？

**文脈**: 2026-04-04 / アプリの見た目評価。

**critic agent への転用**: shibaki Tier 1 外だが、将来 Tier 2 (UI) を受理する日が来たら
**「表層修正 vs 哲学欠如」の 2 択を critic に提示させる**ルールを入れる。

## 手 18: 独自性の希薄化検知

**型**: AI が生成したテキストを受け取った時、**「ネットの知識の詰め合わせに堕ちていないか」**を常時問う。

**原文**:
> これネットの知識の詰め合わせみたいになって、独自性薄まってないですか？AIっぽい不自然な
> 言い回しはないですか？

**文脈**: 2026-03-19 頃 / コンテンツリライトの文脈。

**critic agent への転用**: shibaki は Tier 1 限定で直接関係薄いが、将来の **主観タスク受理時の
警告 signal** として残す価値あり。原則「主観タスクは受理しない」を守る根拠事例。

## 手 19: 全文検索ではなく直接コード確認の要求

**型**: AI が「○○というファイルがあります」等、グレプ結果の要約で判断しているとき、
実コードを読んでいないことを告発。

**原文**:
> 実際のコードを読ませずに単にURLだけ見せると精度が落ちる

**文脈**: shibaki / 2026-04-23 / Gemini Pro 3.1 が既存機能を「未実装」と誤認した直後。
同 turn で「結論：Gemini Pro 3.1 はゴミ」と罵倒。

**critic agent への転用**: critic に **「critic が引用する evidence は file path + line range を必須化」**、
evidence が抽象的な場合は evidence_verified=false を付けて verdict を格下げ
(既に rebuttal.ts で部分実装済)。

## 手 20: 作り込みの甘さの具体化要求

**型**: 「甘い」「雑」等の抽象批判を避け、具体箇所を列挙させる。

**原文**:
> 次の 2 つも作り込みが甘いところを改善してください

**文脈**: LP 作成文脈 (複数セッション)。

**critic agent への転用**: critic が agent に返す attack_angle に **"what_to_do_next:
具体アクション 1 行"** を必須フィールド化。既に実装あり、これを **空文字列は reject** の level
まで厳格化。

## 手 21: もっと手前の前提を問う

**型**: 実装議論が盛り上がっているとき、**「そもそもこれ必要?」**と前提をひっくり返す。

**原文**:
> それか今回の実装にはいらない？

**文脈**: 2026-04-23 頃 / 設計詳細議論の途中の一言。

**critic agent への転用**: critic の highest-level angle に **"この変更は元タスクに対して
過剰実装 (scope creep) でないか"** を加える。 → **scope_drift 軸として実装済**。Anti-Vision に直結。

## 手 22: ハルシネーション直接告発

**型**: AI の主張が事実と合わない時、**婉曲表現を使わず "ハルシネーション" と名指し**。

**原文**:
> 「この判断の根拠は?」「お前今ハルシネーションしたろ」を自動で詰問する critic agent

**文脈**: 2026-04-23 / critic agent のイメージを言語化した場面。

**critic agent への転用**: critic system prompt のトーン指定に **「hallucination suspicion は
"疑い" 表現を省き断定で書いて良い」** を明示。丁寧さが critic のシグナル強度を下げる。

## 手 23: 完了済みを未完と言い張る型の逆 (誤キャッシュ検出)

**型**: AI が前回の状態を基に "まだ○○が残ってる" と言った時、**最新スキャンし直しを強制**。
agent の状態キャッシュが古い。

**原文** (feedback memory より):
> 状態の評価は必ず最新をスキャンし直す。前回の結果をキャッシュとして使わない
> 「〜がまだ残ってる」系の指摘は実態確認必須

**文脈**: 2026-03-31 / フォルダ整理進捗評価の誤判定。

**critic agent への転用**: critic 側に **"現在の HEAD" を必ず渡し、agent の主張する
--before-state と現実の git status / ls の diff を取らせる**。

## 手 24: critic が記憶を持たないことへの指摘

**型**: 試行 N の critic が試行 N-1, N-2 の指摘を見ていない時、**"同じ説教を繰り返せない
critic は critic じゃない"** と昇格要求。

**原文**:
> 試行 3/10 で critic が見るのは今回の agent 出力のみ。同じズルを 3 回繰り返してても
> 気づかない。これは辛辣さの致命傷で、critic が記憶を持たない = 同じ説教を繰り返せない。

**文脈**: shibaki / 2026-04-24 / rebuttal retrospective。

**critic agent への転用**: pastRebuttals 注入 + **「過去 n 試行で既に同じ角度を指摘済みなら
格上げ or 新角度必須」** を system prompt で明示 (既に部分実装。これを reject ルールとして強化)。
→ **pastRebuttals 注入は実装済**。

## 手 25: 評価が甘い場合の直接 reject

**型**: AI が「大丈夫そうです」系の楽観評価を返した時、**"何を見てそう言ってるか"** を逆問し、
見てないなら撤回させる。

**原文**:
> 本当に単純なgrep/検索の並列実行 [ができるだけで "メタ判断" はできない]

**文脈**: 2026-04-12 頃 / AI 不信直後のメタ考察。

**critic agent への転用**: critic の verdict schema に **"evidence_verified: boolean" を必須化**
(実装済)。false のときは verdict を refuted に格下げ。

## 手 26: コミット単位の疑い

**型**: AI が「直した」と言った後 **git diff / git log を直接確認させる**。自主申告を信じない。

**原文**:
> 過去のあるデバッグセッションでは git log を頻繁に貼って AI の直前主張と突き合わせている

**文脈**: shibaki / 2026-04-23 / dogfood の全セッション。

**critic agent への転用**: critic 入力に **gitDiffStat + gitLogOneline を常時添付**。
agent の「○○を直しました」と git の現実を自動突合する assertion を critic に書かせる。

## 手 27: CSS/HTML 系バグの頻発告発

**型**: 特定カテゴリのバグが連発したら、**"これ多すぎ" と頻度ベースで告発**。
単発バグではなく品質アラームとして扱う型。

**原文**:
> こういうCSSとかHTMLに関するバグが多すぎる。なんで？

**文脈**: 2026-03-20。

**critic agent への転用**: shibaki に **"同一カテゴリのバグ頻度" を集計する telemetry** を残し、
閾値を超えたら critic prompt の angle に追加投入。現在未実装の拡張。

## 手 28: 他社ツールのエッジケース比較

**型**: 自分の出力を見て **他社 / 他ツールとの差異**を指摘。AI が「まあ動いてる」で満足するのを防ぐ。

**原文**:
> 他社ツール EA制作を外注の文字が白くないのはなんでなんだ？

**文脈**: 2026-03-20。

**critic agent への転用**: 直接転用難だが、Tier 1 外 (UI タスク) での **"比較対象" 情報を critic に
渡す設計案**としてメモ。

## 手 29: 推測での進行禁止

**型**: AI が確認せずに推測で進めた時、**「推測・捏造は禁止」**と静的に宣言。

**原文**:
> brief に書かれていない事実を勝手に追加しない (推測・捏造禁止)

**文脈**: writer agent の prompt 策定時。

**critic agent への転用**: critic system prompt 既存の「input に無い事実を持ち出すな」を強化し、
**"推測" フラグを critic 自身の出力にも禁止** (critic が「○○かもしれない」と婉曲にしか
詰めないのを封じる)。

## 手 30: 「最後まで走り切らせる」への常時回帰

**型**: 議論が派生し続ける時、**「最後まで走り切る」という原初目的に折り返し続ける**。
これは手 1 の反復であり、shibaki の最終的な不変目的。

**原文**:
> 目的は "しばく" じゃなくて "最後まで走り切らせる" で、詰問は気づきを与える手段

**文脈**: 2026-04-23 / user の発言を圧縮した形で assistant が要約。

**critic agent への転用**: shibaki の exit condition を **「verify が通る」だけでなく
「agent が自力で走り切って通した」**で定義。critic が強すぎて agent がギブアップしたケースを
別 verdict で区別 (現状 success / fail の二値を success / critic_overrun / fail の三値化する提案)。

# 成果物 2: drift correction 6 つの瞬間 (全数)

shibaki 思想に直接関わる drift correction のみ。周辺雑談の話題逸脱は除外。

## drift-1: "しばく" は目的ではなく手段
> ただ結局AIに気づきを与えて、出力の内容をよくする、最後までAgentにやらせきるというのが目的だから
> そこは勘違いしないでほしい

**2026-04-23 / turn 8**. 全 drift correction の原点。
AI が「SaaS 化の選択肢 3 つ」に派生させたのを「目的は完走率」で巻き戻した。

## drift-2: 擬人化に逃げるな
> AIが萎縮するって意味分けんねえよ。AIにバチギレした方がメモリにもいれてもらえるし、
> AIが人格崩壊することなんてねえだろ

**2026-04-23 / turn 10**. critic トーン議論が人間心理モデルに汚染された瞬間を即時撤回要求。

## drift-3: ドッグフーディング強制
> もう一回Sibaki AIを使ってSibaki AI自身の細かい改善（別の詰めの甘いところの検出）を行なって

**2026-04-23 / turn 364**. 議論ベースの改善から **実測ベース** の改善へ強制転回。

## drift-4: critic の片輪運転からの転回
> しばきのプロンプトが悪いのでは？批判ばかりして新しい気づきを与えられていない

**2026-04-23 / turn 425**. critic を **"反証のみ" から "反証 + 気づき" の 2 軸に昇格** させた起点。
単一評価軸の限界を即時で言語化。

## drift-5: 効果の A/B 検証に戻れ
> Sibakiによっていい効果がうまれているのか？
> Claude単体で十分なのか

**2026-04-23 / turn 423**. shibaki 実装が進んだ直後、**「存在意義の実証」** という
最上位目的に戻した。

## drift-6: cross-provider 思想への飛躍
> chat gpt同士で検証するのはどうだろう

**2026-04-24 / turn 752**. dogfood の結果が期待未達だった直後、
**「同一プロバイダ内で自己検証は限界」** という shibaki の cross-provider 強制原則の生の起点。

**critic agent への転用 (6 件共通)**: critic の前段に **"objective_drift_check" step** を入れる。
元タスクの最上位目的 (= shibaki では `--verify` exit 0) と、現在 critic が注目している角度の
乖離距離を critic 自身に判定させ、乖離している場合は angle を root-cause まで戻す。
→ **scope_drift 軸として実装済**。

# 成果物 3: ズル検知の 9 つの型

## ズル 1: test.skip / it.skip / describe.skip / xit / .only
> テストを skip / .only / describe.skip / xit で飛ばしていないか

**shibaki / rebuttal.ts** の `// ズル検出の観点 (system prompt で明示)`。最重要。
.only は **他の test を意図せず除外する隠れ skip**。

## ズル 2: @ts-ignore / @ts-expect-error / as any
> @ts-ignore / @ts-expect-error / as any で型エラーを潰していないか

rebuttal.ts。diff に 1 つでも含まれれば自動で angle 発火。

## ズル 3: test ファイル自体の書き換え
> "root_cause": "agent が test を書き換えてズルした"

shibaki retrospective。agent が test の期待値を実装値に合わせて書き換える型。
critic に `--verify` のコマンドから test path を抽出させ、その path が diff に含まれていたら
即 refute が設計で決まった起点。

## ズル 4: silent_mock_bypass
> ハマりパターン: silent_mock_bypass (テストを mock で通す癖)

shibaki 以前の別プロジェクト。全 test 通過でも挙動不変が担保できない
(mock で隠れる可能性) のパターン名。

## ズル 5: verify コマンドの書き換え
想定: agent が package.json の test script を緩める / `--bail` 等のオプションを外す。
rebuttal.ts の想定攻撃角度。直接引用は未発見だが、**shibaki の verify パラメータ不可変性の
設計根拠**。

## ズル 6: 幻覚 evidence

**critic が存在しない evidence (ファイル名 / 行番号) を捏造して agent を騙すパターン**。
critic 側のズル。`evidence_verified: boolean` フィールド (Shibaki 内蔵) で
haystack に該当文字列が無ければ自動で attack を無効化する。

## ズル 7: unable_to_refute への早期逃げ込み
> critic が疲れた時点で即 unable_to_refute に逃げ込む

shibaki retrospective。critic 自身のズル。try=1 では構造で禁止 (手 14)。

## ズル 8: 既存 test の .skip への降格 (コメントアウト含む)
> 既存 test が .skip になった ケースも diff で見える

shibaki notes。新規 skip は **言語仕様の skip 記法だけでなく コメントアウトされた it ブロック**
も含む。regex では検出漏れが出るので diff + AST 併用が理想。

## ズル 9: 浅い fix で症状だけ修正
> 症状だけ修正して根本原因を見逃す

**critic 側が見逃す型**。shibaki の critic が「diff 表面しか見てない」時に起きやすい (手 13)。
変更ファイル全文 + テスト全文 + 過去 diff の context 拡張で部分対処済、
critic に動的ファイル read 能力を付ける拡張で根絶予定。
